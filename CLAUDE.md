# UsefulDesk — Product Context (India-First Gym CRM)

> Shared context for every session. UsefulDesk is being turned into an **India-first CRM for gyms, boutique studios, and (eventually) franchises**. Market analysis lives in `PRDs/india_gym_crm_pain_points.md`.

> **Keep this file current (rule, non-negotiable).** Every time you land a **major architecture change or a new feature**, update this CLAUDE.md in the *same* change — treat the doc edit as part of "done", not a follow-up. Touch the sections it affects (Architecture, UI reusability, Current status / "Built since", Roadmap To-Do, data model). Keep entries terse and factual: what shipped, where the code lives, any reuse/gotcha a future session must know. Prune stale lines rather than stacking contradictions. Small bug fixes / refactors don't need an entry; anything that changes how the codebase works or adds user-facing capability does.

## The one thing to understand first

This repo (`wacrm` v0.7.0) **looks** like a generic WhatsApp CRM but is **not primitive** — it's a mature multi-tenant WhatsApp CRM. It already ships best-in-class versions of the exact layer the market wants (WhatsApp send + templates, broadcasts, automations, shared inbox, RBAC, public API). What is genuinely missing is the **gym domain** (members, plans, memberships, renewals, payments, attendance).

So the strategy is **not** "build a CRM." It is: **keep the WhatsApp/comms foundation, add a gym domain layer on top, wire the two together.** The renewal + WhatsApp workflow that wins this market is ~60% already built.

## Market insights (from the PRD)

Indian gyms don't lack CRM features — they lack a *simple, reliable, WhatsApp-first, UPI-aware* layer for the messy exceptions (renewals, collections, follow-ups, lapsed members). They fall back to WhatsApp + Excel + phone + paper. Biggest pains, most urgent first:

1. **Payments / renewals / reconciliation** — manual chasing via WhatsApp/UPI/cash/Sheets. Revenue leakage. Most monetizable.
2. **WhatsApp is the real CRM** — should be the core channel, not an add-on.
3. **Reports aren't actionable** — owners want *action lists* ("who to call today"), not dashboards.
4. Reliability/sync failures · heavy onboarding/too many clicks · weak staff mobile workflows · weak family/multi-location plans.

**The wedge:** *Know who's expiring → remind on WhatsApp → collect on UPI → assign follow-up → track the conversation.* Everything else is secondary until this loop works.

## Product principles (non-negotiable)

Phone-first · WhatsApp-native (not add-on) · renewal-first · **action lists over dashboards** · simple-for-boutiques / controlled-for-franchises · offline-tolerant · payments + reminders in one flow · every exception has an owner + status + next action · don't force a member app early · owner feels in control in 30 seconds.

**Every feature must answer:** does it help the owner save time, recover lost leads, collect renewals, or retain members? If not — defer.

## Architecture (how this codebase works)

- **Stack:** Next.js 16 (App Router; middleware is renamed `proxy` — see `src/proxy.ts`), React 19, TypeScript, Supabase (Postgres + Auth + Storage + RLS), Tailwind v4, Base UI / shadcn primitives (`src/components/ui/`), **Motion (`motion/react`) for animation**. No zod (validation hand-rolled). Vitest, colocated `*.test.ts` in `src/lib`.
- **No `"use server"` actions.** Backend = `src/app/api/**/route.ts` handlers OR direct Supabase browser-client calls from client components, secured by Postgres RLS.
- **Tenancy:** tenant = `accounts` row; users belong to one via `profiles.account_id`. RBAC `owner > admin > agent > viewer` (`account_role_enum`) enforced by RLS helper `is_account_member(account_id, min_role)`; mirrored in `src/lib/auth/roles.ts` + `useAuth()` (`canEditSettings`, `canSendMessages`). **Copy the RLS pattern from `supabase/migrations/017_account_sharing.sql`.**
- **Migrations:** sequential + idempotent in `supabase/migrations/` (enum in `DO $$ IF NOT EXISTS` block, `CREATE TABLE/INDEX IF NOT EXISTS`, drop-then-create policies, reuse `update_updated_at_column()`). Latest is `063` (plus one timestamp-named `20260711173414_harden_membership_payments.sql`; keep NEW files sequentially numbered; applied live via MCP; note `supabase db push` fails on a migration-history mismatch with the MCP-applied timestamped versions — always apply via MCP `apply_migration`). The live Supabase project is `UsefulDesk` (`fwqthstqrkrwtaehefks`) — new migrations are applied to it via the Supabase MCP `apply_migration` (or `supabase db push`); after applying, verify with a `pg_policies`/schema query.
- **Localization (migration `055`, non-negotiable pattern):** every gym (account) carries its own regional config on `accounts` — `country_code, locale, timezone, date_order ('DMY'|'MDY'|'YMD'), time_format ('12h'|'24h'), week_start (0|1|6), phone_country_code, measurement_system` + the pre-existing `default_currency`. **Geography lives ONLY in `src/lib/locale/config.ts`** (`COUNTRY_PRESETS` — one row per country; adding a country = adding a preset). `resolveAccountLocale(row)` narrows the account row (field-by-field India fallback) into an `AccountLocale`; `buildFormatters(cfg)` (`src/lib/locale/format.ts`) is the ONE formatting surface — `fmt.date/dateShort/time/dateTime/number/money/moneyShort/today()`. Client: `useLocale()` (`src/hooks/use-locale.ts`) returns the AuthProvider's memoized `{ locale, fmt }` (AccountSummary in `use-auth.tsx` selects the columns; `refreshProfile()` re-resolves after a settings save). Server/cron: `resolveAccountLocale(accountRow)` + `buildFormatters`. **Never** hand-roll `toLocaleDateString`/`toLocaleString`/`Intl.NumberFormat`/`format(date-fns)` for gym-domain output, never write `'en-IN'`/`'Asia/Kolkata'`/`if (country === …)` at a call-site. Timezone helpers for instants: `todayInTz`, `hourInTz`, `dayStartInTz` (day-start queries), `dateAtNoonInTz` (stamping picked days into timestamptz, e.g. `payments.paid_at`), `timeInTzToUtc`/`hhmmInTz` (reminder slots). `istToday()` survives only as the India-default fallback for pure-lib `today` params — components always pass `fmt.today()`. Signup has a country picker → preset → `auth.signUp` metadata → `handle_new_user` (055, regex-guarded) writes the columns; Settings → Localization (`localization-settings.tsx`) edits everything (country change re-applies the preset; live preview). UPI is gated by `upiAvailableFor(currency)` (INR-only rail — a currency condition, not a geo conditional). CSV-import ambiguous dates default to `importDateOrder(locale)`. Scope = regional formatting, NOT string translation.
- **Authored-content ownership (rule):** notes (and any future authored content — comments, activity remarks) belong to their author. Only the author may edit; only the author **or admin/owner** may delete (moderation). Enforced in BOTH layers: RLS (`user_id = auth.uid() AND is_account_member(account_id,'agent')`, plus `OR is_account_member(account_id,'admin')` on delete — see `046_contact_notes_ownership.sql`) and UI (gate the affordance via a `roles.ts` predicate, e.g. `canDeleteAnyNote`). Never a UI-only gate.
- **Role capabilities:** every "who can do X" check is a named predicate in `src/lib/auth/roles.ts` (+ test in `roles.test.ts`), mirrored by the RLS policy. Adding a capability = one predicate + one policy — no inline role comparisons at call-sites.
- **Supabase RLS-blocked writes are silent:** `.delete()`/`.update()` blocked by RLS return **no error, zero rows**. For destructive ops, chain `.select('id')` and treat an empty result as failure — otherwise the UI toasts success while the row survives (see `deleteNote` in `contact-detail-view.tsx`).
- **Reuse, don't rebuild:** WhatsApp send = `sendMessageToConversation` (`src/lib/whatsapp/send-message.ts`); dashboard send route `POST /api/whatsapp/send` accepts `contact_id` + `message_type:'template'` and **find-or-creates the conversation** (reaches members with no thread). Contact dedupe = `findExistingContact` / `isUniqueViolation` (`src/lib/contacts/dedupe.ts`). Media upload = `uploadAccountMedia('chat-media', file)` (`src/lib/storage/upload-media.ts`). All regional formatting (currency, dates, times, numbers, "today") = the account locale layer — `useLocale().fmt` client-side, `buildFormatters(resolveAccountLocale(row))` server-side (see the Localization bullet); `formatCurrency(value, currency, locale?)` (`src/lib/currency.ts`) underlies `fmt.money`.
- **Lint gotcha:** repo enforces `react-hooks/set-state-in-effect`. Never call a setState-wrapping function directly in `useEffect`. Load data with an inline `(async () => { … })()` IIFE + a `cancelled` guard; expose manual refetch as a nonce bump. See `src/components/members/*`.

## UI component reusability (non-negotiable)

- **One header per page — title, actions, and sub-nav tabs all live in the shared app bar** (`src/components/layout/header.tsx`). A page must **not** own a second title/subtitle row or a standalone tab strip. The header is a two-row `flex-col` with the divider on the outer `<header>`: **row 1** = route title (from the `pageTitles` map) + a trailing actions slot; **row 2** = a tab slot (`empty:hidden`, so tab-less pages stay a single title row). Pages portal chrome in via `page-header-actions.tsx`: `<PageHeaderActions>` (primary/secondary buttons — Import / Export / Add …, next to the title; gate with `GatedButton`'s `canAct` + `gateReason`, don't hide) and `<PageHeaderTabs>` (sub-navigation). Because the divider is on the whole header, a filled tab row makes the divider fall **after** the tabs, so the nav reads as part of the header. Tabs use `ui/tabs.tsx` **`variant="line"`** (underline style) driven controlled (`value`/`onValueChange`) — NOT a pill segment bar. Canonical example = `/leads` (actions) + `/members` (actions **and** line tabs, replacing its old `bg-muted` segment toggle). Every new page does the same. **The line variant's active tab is `--primary`** (label + underline; Jul 2026 — was `foreground`/near-black, which ignored the account's chosen accent theme, so a Cobalt/Emerald/Amber gym still got a black underline). It's a **master** change in `ui/tabs.tsx`, so every line-tab surface moved together (members sub-nav + the member-sheet jump nav); the default/pill variant is untouched (its active state is the raised white chip, not a tint). A tab strip that can overflow must also **keep the lit tab in view** — a scrollspy-driven nav (member sheet) centres the active tab in the strip on change, else on mobile it lights a tab parked off-screen and the nav reads as having no active section.
- **Never hand-roll a UI element that exists in `src/components/ui/`.** Before writing any pill/button/input/dialog markup, check for a primitive there and use it. If no primitive fits, **stop and ask the user**: create a new master component, or use a different existing one? Don't silently roll an inline one-off.
- **Master components are single sources of truth.** A change to a file in `src/components/ui/` changes every usage across the product — **warn the user before editing one** and list what it affects. Restyling a single call-site goes through `className`/variants, never by forking the component.
- **No inconsistency in UI pattern tokens (non-negotiable).** Sibling components that read as the same *kind* of thing must share the same tokens — the popup padding, the border/ring, the radius, the item inset, the icon size, the muted-fill. Before you add or edit any component, open the **closest matching existing one** and copy its tokens verbatim; don't invent a value or eyeball it. Drift here is a real bug, not a nit: e.g. `DropdownMenuContent` puts `p-1` on the popup, but `SelectContent` had none — so a bare-item `ui/select` (no `SelectGroup`) rendered its items flush to the popup edge while a dropdown of the same shape looked padded (fixed by moving `p-1` onto the Select's `List`). When you notice such a mismatch anywhere, fix it at the master component so every call-site converges, and cross-check the peer components (menu ⇄ select ⇄ combobox ⇄ popover; the search field, the badge/pill family, the segmented toggle) for the same divergence.
- **Cursor (Jul 2026, base rule — don't re-add per component):** Tailwind v4's Preflight sets `button { cursor: default }`, so every button in the product (Renew, Add member, tabs, dropdown/select triggers, icon buttons) rendered an **arrow** while links, checkboxes, switches and clickable table rows rendered a **hand** — the drift came from fixing it one call-site at a time. One base rule in `globals.css` now owns it: `button:not(:disabled), [role="button"]:not(:disabled) { cursor: pointer }` (a `:disabled` control keeps the arrow — a dead affordance must not advertise itself as clickable). Never add `cursor-pointer` to a button/tab/trigger again; a **non-button** clickable (a `<div>`/`<tr>` row, a card) still needs it explicitly.
- **Person avatars:** the canonical avatar is `UserAvatar` (`src/components/ui/user-avatar.tsx`) — photo when `src` (a `profiles.avatar_url` / future member photo URL) is set, first-initial fallback on the primary tint otherwise. **Every** person render (teammates in notes/rosters/sidebar, and future members/contacts in table or board views) goes through it so a photo uploaded once appears everywhere; never hand-roll `Avatar + AvatarImage + AvatarFallback` for a person again. Size via `size`/`className`, initial restyle via `fallbackClassName`, badge overlays (presence dots) as children. Teammate avatar URLs come from `useAccountStaff()` (`avatarById` map); the current user's from `useAuth().profile.avatar_url`.

Used in **every** member row/column so a person reads identically across the all-members table, renewals tables, follow-ups, trials, payment-due, inactive lists, check-in, and the payments ledger; never hand-roll a bare name+phone stack for a member again. Optional `meta` node = a third context line the caller styles (e.g. "plan · due date"); when `meta` is present the avatar top-aligns, else it centres on the name+phone pair. `src` = the member photo (`contacts.avatar_url`) — pass it at every call-site so an uploaded photo shows everywhere.

- **Search fields:** the canonical search bar is `SearchInput` (`src/components/ui/search-input.tsx`) — a leading search glyph over a **rounded-rectangle** `Input` with the `border-border` token + muted fill. Width/flex go on the wrapper via `containerClassName` (e.g. `max-w-xs flex-1`); the radius/border/icon/padding are fixed — never restyle those per call-site (that drift — leads was a `rounded-full border-border` pill, members a `rounded-lg input-border` box — is exactly what this killed). Used by the leads + members + check-in toolbars, the inbox conversation list, and the import / manage-columns dialog pickers. The `Combobox`'s in-popover search and `global-search`'s command-trigger are deliberately their own patterns, not this.

Use it for lists too long to scan (e.g. the import wizard's field picker); short static lists stay on `ui/select`. Don't hand-roll popover+input search UIs.
- **Form fields are unfilled — never add a `bg-muted` fill (Jul 2026, non-negotiable):** every form control (`Input`, `CurrencyInput`, `Textarea`, `SelectTrigger`, `DatePicker`, and the rare native `<input>`) renders on the primitive's own `bg-transparent` (+ `dark:bg-input/30`) — a bordered, unfilled field. It reads as a field because of `border-input-border`, not because of a grey box. Call-sites had been hand-adding `bg-muted` (the `bg-muted border-border text-foreground placeholder:text-muted-foreground` / `w-full bg-muted` idioms), which painted a grey slab inside every dialog and settings form; **all ~180 were stripped** across auth / contacts / leads / members / settings / broadcasts / automations / flows. Don't reintroduce it on a new field, and don't "fix" a field that looks too plain by filling it. `bg-muted` remains correct and untouched for **non-field** surfaces — decorative containers and summary/info boxes (`bg-muted/20`–`/40`), pills/badges/chips, avatars, icon boxes, code chips, table headers, skeletons, message bubbles, segment toggles, the Calendar's "today" cell — and for **state** styles (`hover:` / `focus:` / `data-[…]:bg-muted`). Deliberate exceptions that KEEP a muted fill (they're a different pattern, not form fields): `SearchInput` (`ui/search-input.tsx` — its muted fill is the canonical search look) and the chat surfaces (inbox `message-composer`, `contact-sidebar` note box, `ai-playground` — filled chat bars).
- **No native `<select>`, ever (Jul 2026):** every dropdown picker in a form is the design-system `Select` (`ui/select.tsx`) — native selects render an unstylable OS popup that clashes with every other popup in the product, and their hand-rolled trigger classnames drifted from `Input`'s tokens (h-9 vs h-8, `border-border` vs `border-input-border`). All ~40 hand-rolled `<select>`s were converted; a short-lived `NativeSelect` master was created and then superseded/deleted in the same pass. The working idiom (see the gender picker in `member-personal-info.tsx`): `<Select value={x || undefined} onValueChange={(v) => set(v ?? "")}>` + `<SelectTrigger id={…} className="w-full"><SelectValue placeholder="…" /></SelectTrigger>` — trigger defaults to `w-fit` so pass `w-full`; **no `bg-muted`** (see the form-field fill bullet); `id` lives on the trigger (keeps `<Label htmlFor>`); `disabled` on the root. Gotchas: Base UI types `onValueChange` as `string | null` (guard always-set handlers with `(v) => v && f(v)`); an optional/clearable field gets a `<SelectItem value={null}>` first item (null re-shows the placeholder — a *selected* null item ALSO renders as placeholder, so a real option mapped to `""` state, like contact-form's "New" status, uses `value={null}` + a dynamic placeholder); `<optgroup>` ⇄ `SelectGroup`+`SelectLabel`. **Controlled-vs-uncontrolled (Jul 2026 fix):** Base UI latches `isControlled = value !== undefined` into a **ref on the FIRST render** (`@base-ui/utils/useControlled`), so the `value={x || undefined}` idiom above made every Select mount *uncontrolled* (empty state → `undefined`) and flip to *controlled* on the first pick — a dev-console warning ("A component is changing the uncontrolled value state of Select to be controlled"), and worse, an uncontrolled root ignores a programmatic reset (form cleared, dialog reopened on another record). `null` is the correct "controlled, nothing picked yet" value; `undefined` means uncontrolled. Rather than rewrite the ~12 call-sites and leave the trap armed for the next one, the **`Select` wrapper coerces an explicitly-passed `value: undefined` to `null`** (keyed on `"value" in props`, so a genuinely uncontrolled Select — one that passes no `value` key and uses `defaultValue` — is untouched). The `value={x || undefined}` idiom therefore stays correct at call-sites; don't "fix" one by switching it to `defaultValue`. **Trigger label resolution (Jul 2026 fix):** Base UI's `Select.Value` renders labels ONLY from the root's `items` prop — never from mounted `SelectItem` children — so a selected value used to echo raw (plan UUID, `male`). The `Select` wrapper in `ui/select.tsx` now auto-derives `items` by walking its JSX children for `SelectItem`s (explicit `items` prop wins; null-valued items skipped so the clearable-null-item placeholder idiom keeps working). Caveat: `SelectItem`s hidden inside a custom component (not directly in the Select's JSX subtree) aren't seen — those call-sites pass `items` explicitly. Long searchable lists still go to `Combobox`; custom-rendered option rows (pills, avatars) to `ui/select` or a `DropdownMenu` picker per the leads bulk-dialog note.
- **Date fields (Jul 2026):** the canonical date field is `DatePicker` (`src/components/ui/date-picker.tsx`) — a **whole-field-clickable** Input-styled Popover trigger (CalendarIcon + `fmt.date` display) opening `Calendar` (`src/components/ui/calendar.tsx`, react-day-picker v10). Locale-aware: display via `useLocale().fmt.date`, `weekStartsOn = locale.weekStart` (the first real `week_start` consumer). `Calendar` gotchas: month/year caption dropdowns route through **`ui/select`** via a custom `components.Dropdown` (rdp only reads `Number(e.target.value)` off the change event, so a `{ target: { value } }` synthetic is handed to it — do NOT let rdp render its native `<select>`); day cells/nav reuse `buttonVariants`; no rdp CSS import (all classNames overridden, shadcn-style). Dropdown year range defaults to `min`→`max` else −100y→+5y via `startMonth`/`endMonth`. Layout is **fixed-width** (`root w-[16.5rem]`, month/year triggers `months_dropdown`/`years_dropdown` fixed widths, day cells `flex-1`) so the popup never resizes with the month name; the rdp nav strip is `pointer-events-none` (chevrons re-enable) or it paints over and swallows the caption-dropdown clicks. A **"Today" footer link** (account-tz `fmt.today()`, hidden when outside `min`/`max`) picks today and closes.
- **Data-grid column header:** the per-column header UI (label + single double-sided sort toggle + three-dot overflow menu carrying Sort ascending/descending, an Excel-style value **Filter** submenu, and column actions) is the shared `ColumnHeader` (`src/components/table/column-header.tsx`) — the single source of truth for both the `/leads` table and the All-members table. Generalized: freeze / add-column / edit-options / drag-handle / the greyed "smart property" placeholder are all **optional props**, so leads passes the full set and members mounts just Sort + Filter + Hide. Sort is ONE `ChevronsUpDown` button that cycles asc→desc (saves header width; shows the active direction lit). The resize grip + any drag transform live on the OWNING `<th>`, never in `ColumnHeader`. Leads' `HeaderCell` is now a thin adapter over it; members renders it directly. Don't fork it — restyle via props. - **Lead-field cells:** status pill / assignee chip / editor option lists live in `src/components/leads/lead-cell-renderers.tsx` (`StatusBadge`, `AssigneeDisplay`, `statusCellOptions` / `sourceCellOptions` / `genderCellOptions` / `assigneeCellOptions`, `customEditKind`) — consumed by BOTH the `/leads` table and the import preview grid so they can't drift. New surfaces rendering lead fields go through these.
- **Animation (Motion):** animate through the shared primitives in `src/components/ui/`, not scattered `motion.*` at call-sites. Canonical set: **`Collapse`** (`collapse.tsx`) — `open`-driven height+fade reveal that unmounts when closed (replaced the bulk toolbar's `grid-rows-[0fr↔1fr]` hack; the surrounding flex gap closes on its own); **`MotionList` + `MotionListItem`** (`motion-list.tsx`) — wrap a `.map` so items fade/slide in-out and FLIP-reflow on add/remove/reorder (`AnimatePresence mode="popLayout"`, `layout`; used on the notes list + `/notifications`); **`AnimatedNumber`** (`animated-number.tsx`) — count-up on scroll-into-view, drives text via ref (no per-frame re-render), honours reduced-motion (used in dashboard KPI tiles; pass `format` for currency). The **kanban board** (`leads-board.tsx`) uses `LayoutGroup` + per-card `layout="position"`/`layoutId` so a dragged card *flies* to its new column — dnd-kit owns the drag; its `DragOverlay dropAnimation` is `null` so Motion's FLIP owns the settle (don't re-enable it — double-animates). **The FLIP only stays smooth because every card is `contain: layout` (`[contain:layout]`)** — Motion flushes layout once per move to measure all cards, and containment makes each card an isolated layout subtree so that flush skips the card's ~30 internal nodes (avatar/SVG/badges/dropdown) and only re-lays-out the column's boxes. Without it, the rich card body froze the drop at scale (this bit us — the fix is containment, NOT stripping the animation). See the board-parity "Drag perf" note for the full render structure. **Gotchas:** (1) **Base UI dialogs/dropdowns/sheets/popovers already animate** via `data-open/closed` + `data-starting/ending-style` (tw-animate-css) — do NOT wrap them in Motion; it fights their mount lifecycle. (2) **Never put a `motion.*` (transform) on a `<tr>`/ancestor of the leads table** — a transformed ancestor becomes the containing block and **breaks `position: sticky` frozen columns**. Row enter/exit is intentionally not animated for this reason.
- **Money inputs:** the canonical amount field is `CurrencyInput` (`src/components/ui/currency-input.tsx`) — the master `Input` with the account's currency symbol overlaid (`currencySymbol(locale.currency)`). Two modes: plain (`type="number"`, `value`/`onChange` — the original 3 lead call-sites) and **grouped** — pass `groupLocale={locale.locale}` + `onValueChange` and it renders `type="text"` with locale grouping as you type (`₹1,00,000` on en-IN) while still handing back the RAW numeric string; caret is restored by digit position so separators don't kick it to the end. A `type="number"` field can NEVER show separators — any new money field that should group uses the grouped mode (plan editor's price does). **Gotcha:** an `overflow-y-auto` scroller (dialog bodies) or a `Collapse` clips the focus ring on BOTH axes (a non-`visible` `overflow-y` forces `overflow-x: auto`) — give it `-mx-1 px-1 py-1`-style inner padding or every field inside gets its ring sliced.

Wrap the money part in a `<span className="tabular-nums">` when the surrounding text is prose. - `membership_periods` (migration `057`) — **one row per billing cycle = one invoice** (`period_start/end`, `fee_amount` snapshot, `state: open|void`, `UNIQUE(membership_id, period_end)`). The `memberships` row stays the **current-cycle pointer** (its start/end/fee mirror the live cycle, so every existing read keeps working); periods accumulate the HISTORY so recurring members get a real Paid/Unpaid/Upcoming trail + arrears. **View `membership_period_invoices`** adds `amount_paid`/`balance` by reconciling payments on `period_end` (the same trick as `membership_dues`, so **no `period_id` on payments**). Paid/Unpaid/Upcoming status is **derived in TS** (`periodStatus()` in `src/lib/memberships/periods.ts`) — it needs the account's tz "today", so it's kept OUT of SQL (view stays tz-agnostic). **Lifecycle:** an `AFTER INSERT` trigger on `memberships` (`create_initial_membership_period`) auto-creates the first period → all 5 create paths need **zero TS**. renew/convert go through `renew_membership_transaction` (harden migration), mid-cycle plan swap/upgrade through `change_membership_plan` (061 — truncates the current period at the switch date, re-invoices it at the pro-rated used value, re-stamps its payments under the 058 GUC, opens the new plan's period, rolls the pointer, optional first collection; credit math is the pure `planChangeQuote` in `src/lib/memberships/plan-change.ts`, tested), and edit/unfreeze/cancel/reactivate through the 058 lifecycle RPCs (`edit_membership_cycle` / `unfreeze_membership` / `set_membership_cancellation`, thin TS wrappers in `periods.ts`) — a trigger can't tell a renewal from an edit or an unfreeze, and each RPC is one transaction. Renew opens a NEW period (old one stays = arrears); settling an old arrears invoice records against **that** period (RecordPaymentDialog gained an optional `period` prop) and does NOT touch the membership's live `fee_status`. Operational (agent write), realtime-published (054-style). **Reconcile-key gotcha (load-bearing):** because payments↔periods join ONLY on `period_end`, any op that moves a cycle's `period_end` (unfreeze pushes it by the frozen days; an edit can shift it) MUST re-stamp that cycle's existing payments to the new key or they orphan and a paid cycle reads Unpaid — the 058 RPCs do this INSIDE the same transaction (and `payments.period_start/period_end/plan_id` are protected financial fields: a direct agent update is rejected by `protect_payment_financial_fields`, so the RPCs are the only re-stamp path besides an admin). Lifecycle RPCs raise real errors (no silent-RLS ambiguity); the remaining direct membership write (freeze) still chains `.select('id')`. The single *next* Upcoming invoice is **projected in TS** (`projectNextInvoice`, display-only — can't be real until it happens); it returns null for a lapsed membership (`end_date <= today`) so an **expired** member never shows a phantom past-dated "Unpaid" projection. Past/current are persisted. `InvoiceDetailDialog` reads the view's `amount_paid` (not `fee − balance`) so an over-paid cycle's total is honest. **Display-precision money rule (Jul 2026, non-negotiable):** `formatCurrency` renders **0 fraction digits**, so any amount below half a currency unit prints as `₹0` — and a mid-cycle plan change routinely leaves such a residue (it re-invoices the truncated cycle at its pro-rated used value: a one-day stub bills ₹0.32). A raw `balance > 0` test therefore put a **"Due" pill on a row reading ₹0 / ₹0 / ₹0**, and the money is unchaseable anyway (the ledger's ≤-balance guard rejects even a ₹1 payment against it). Every money judgement now goes through **`isChargeableAmount(amount)`** (`periods.ts`, `SETTLED_BALANCE_EPSILON = 0.5`) — never `> 0` / `<= 0` on a fee/balance again. The invoice's **payment axis** (orthogonal to its Current/Past/Upcoming/Void lifecycle) is the pure `invoicePaymentState()` → `paid | due | **no_charge**`, rendered by `InvoicePaymentBadge` (`membership-status-badge.tsx`): *No charge* (neutral) = the cycle billed AND collected nothing (a zero-fee cycle, or a stub whose fee rounds to zero) — neither Paid (no money moved) nor Due (nothing owed); a stub that DID collect money still reads Paid. `periodStatus()` delegates to it (`InvoiceStatus` gained `no_charge`), and the same epsilon gates `isCollectiblePeriod`, the Record-payment/Copy-UPI affordances, the member card's "₹x due" chip, the payment-due buckets, and bulk record-payment.

All date math is **IST-first** (`src/lib/memberships/expiry.ts`) — members must not expire a day early/late for a UTC+5:30 owner. "Expired" is derived (no cron needed for M1).
